## Task 3: Системный агент с query_api

### Новый инструмент: query_api
В Task 3 я добавил инструмент `query_api`, который позволяет агенту общаться с бэкендом.

**Параметры:**
- `method` — HTTP метод (GET, POST)
- `path` — путь к эндпоинту (например, `/items/`)
- `body` — тело запроса (опционально, для POST)

**Аутентификация:**
- Используется ключ `LMS_API_KEY` из файла `.env.docker.secret`
- Базовый URL бэкенда берётся из `AGENT_API_BASE_URL` (по умолчанию `http://localhost:42002`)

### Как агент выбирает инструменты
В системном промпте я объяснил агенту правила:
- **Вопросы по документации** (как что-то сделать) → `list_files` / `read_file` из wiki
- **Вопросы о данных** (сколько предметов, статус) → `query_api`
- **Вопросы о коде** (как реализована функция) → `read_file` для `.py` файлов

## Task 3: Системный агент с query_api

### Архитектура
В Task 3 я добавила инструмент `query_api`, который позволяет агенту обращаться к бэкенду. Теперь агент умеет:
- Читать файлы из wiki (через list_files/read_file)
- Запрашивать данные из API (через query_api)

### Как работает query_api
Функция принимает метод (GET/POST), путь и опциональное тело запроса. Для аутентификации используется ключ `LMS_API_KEY` из `.env.docker.secret`, а базовый URL берётся из переменной `AGENT_API_BASE_URL` (по умолчанию http://localhost:42002).

### Как агент выбирает инструменты
В системном промпте я объяснила Qwen правила:
- Вопросы про документацию → list_files/read_file
- Вопросы про данные (сколько предметов, статус) → query_api
- Вопросы про код → read_file для .py файлов
